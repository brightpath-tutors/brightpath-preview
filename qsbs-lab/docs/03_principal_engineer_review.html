Principal Engineer Design Review — Pre-Implementation Audit

This document challenges the existing architecture before a single line of application code is written. It does not redesign anything. It identifies weaknesses, proposes targeted fixes, and produces a prioritized list of improvements that eliminate the most common sources of future technical debt.

Review Overview
Executive Summary
The architecture is fundamentally sound. The decomposition is appropriate. The dependency direction is correct. The most significant gaps are in error handling standardization, result contract consistency, and test architecture. None of these require redesign — they require formalization before coding begins.

Review Area 1
Module Boundaries
F-04 M9 (Scenario Manager) Has Dual Responsibility MEDIUM
M9 is specified as handling 400-scenario CRUD, search, filter, sort, autosave, version snapshots, bulk operations, comparison view rendering, and the detail form. This is two distinct jobs: data management and UI rendering. At the projected 700+ lines, it will be the largest single module and the hardest to test. If the storage layer changes, the UI code is unnecessarily impacted.
Fix: Split M9 into two files. js/modules/scenario-store.js handles all CRUD, search, filter, autosave, versioning, and bulk operations. It has no DOM code. js/modules/scenario-ui.js handles list rendering, detail form, comparison view. scenario-ui.js imports scenario-store.js. Both are independently testable. scenario-store.js can be tested in Node.js without a browser.
F-11 CE3 vs M4 Boundary Ambiguity LOW
CE3 (Valuation calc engine) and M4 (valuation-engine.js, the original module) overlap. The addendum says M4 "is the UI module that collects valuation inputs and displays charts" and CE3 "is the pure calculation engine." But valuation-engine.js in the original module list includes both calcWeightedEV() and calcAggregateGrossAssets() as exports. It is not clear which file owns which function after CE3 is added.
Fix: Explicit rule: any function that computes a number goes in CE3. Any function that renders HTML, manages a Chart.js instance, or reads/writes storage goes in M4 (valuation.js). calcAggregateGrossAssets() moves entirely to RE3/CE3. calcWeightedEV() moves to CE3. M4 imports CE3 and calls those functions. No calculation logic lives in any file under js/modules/.

Review Area 2
Public Interfaces
F-06 Inconsistent Parameter Ordering Across Engines MEDIUM
Current parameter patterns vary: evaluateQSBS(scenario), evaluateAGA(asset_record, law_version, controlled_group_assets, predecessor_assets), calcStockSaleTax(scenario, exclusion_limit, exclusion_pct), evaluateHoldingPeriod(acquisition_date, exit_date, law_version). Some engines take a flat scenario object, some take extracted fields, some take pre-computed sub-results. This inconsistency will cause bugs when calling engines in sequence.
Fix: Adopt one rule: all engines accept (scenario, options) where scenario is the full scenario object and options is an optional object for pre-computed sub-results or overrides. Engines extract what they need from scenario internally. This eliminates parameter ordering bugs and makes function signatures predictable. Example: evaluateAGA(scenario, options) where scenario.asset_record and scenario.law_version are extracted internally. Pre-computed sub-results passed as options.agaResult to skip recomputation.

Review Area 3
Shared Result Contracts
F-01 No Universal Engine Result Contract HIGH
The base docs define a result contract for evaluateQSBS(). The addendum defines an AuditTrail shape. But the two are not formally unified. Some engines return pass/fail tests, some return calculations, some return explanations. RE8 assembles them all, but if each engine returns a slightly different shape, RE8 becomes a complex and fragile adapter. This is the most important contract to standardize before implementation.
Fix: Define one EngineResult contract that all engines return. Engines that don't produce some fields set them to null or empty array. This makes RE8 trivial — it just reads the same fields from every engine output.

Universal EngineResult Contract

Everything in data{} is engine-specific and only read by the orchestrators that know which engine produced it.

Review Area 4
Error Handling
F-02 No Defined Error Handling Strategy HIGH
The architecture documents never define: when does an engine throw vs. return an error state? What happens if gross_proceeds is -5000000? What happens if acquisition_date is "banana"? What happens if the user edits a locked scenario? Without a defined policy, every developer will make their own choice, producing inconsistent behavior.
Fix: Define the policy once in shared/errors.js and apply it everywhere.

Error Handling Policy

Rule 1: Engines Never Throw

All engine functions catch all exceptions internally. Any exception is converted to an EngineResult with status="fail", errors=["Internal error: " + message], and all other fields at safe defaults. The application never crashes due to an engine error.

Rule 2: Invalid Inputs Return insufficient_information

Null, undefined, NaN, or out-of-range inputs produce status="insufficient_information" with missing_inputs[] populated. Engines never return garbage results from garbage inputs.

Rule 3: Impossible Calculations Return Bounded Results

If a calculation would produce Infinity, -Infinity, or NaN, the engine returns 0 for the value and adds an entry to warnings[] explaining what happened. Never expose Infinity in the UI.

Rule 4: Conflicting Assumptions Are Warnings Not Errors

If two inputs are in logical conflict (exit_date before acquisition_date), add to warnings[] and attempt the calculation anyway with the note "inputs may produce unreliable results." The user sees the warning and can fix their inputs.

// js/shared/errors.js
export class EngineError {
  static invalidInput(field, received, expected) {
    return {
      status: "insufficient_information",
      missing_inputs: [`${field}: expected ${expected}, received ${received}`],
      errors: [],
      value: null,
      // ... rest of universal contract at safe defaults
    };
  }
  static internalError(engine_id, message) {
    return {
      status: "fail",
      errors: [`${engine_id}: Internal error — ${message}`],
      // ... safe defaults
    };
  }
  static conflictingAssumptions(field1, field2, message) {
    return {
      warnings: [`Conflicting inputs: ${message}`],
      // continue with calculation
    };
  }
}

// Usage in every engine:
export function evaluateAGA(scenario, options = {}) {
  try {
    const cash = scenario?.asset_record?.cash;
    if (cash === null || cash === undefined) {
      return EngineError.invalidInput('asset_record.cash', cash, 'number >= 0');
    }
    // ... engine logic
  } catch (e) {
    return EngineError.internalError('re3-aga', e.message);
  }
}

Review Area 6
Data Contracts
F-05 Shallow Object.freeze() on lawConfig MEDIUM
Object.freeze() is shallow. lawConfig.lawVersions is frozen but lawConfig.lawVersions.post_july4_2025 is a separate object reference that is NOT frozen. An engine could accidentally mutate lawConfig.lawVersions.post_july4_2025.grossAssetsThreshold = 0 and corrupt all subsequent calculations in that session without any error.
Fix: Implement deepFreeze() utility in shared/utils.js and apply it to lawConfig at export. 
function deepFreeze(obj) {
  Object.getOwnPropertyNames(obj).forEach(name => {
    const value = obj[name];
    if (typeof value === 'object' && value !== null) deepFreeze(value);
  });
  return Object.freeze(obj);
}
export const lawConfig = deepFreeze({ ... });
F-08 No IndexedDB Schema Version Migration Strategy MEDIUM
IndexedDB schemas are versioned. If the schema changes (adding a new store, adding an index, adding a column), the IDB version must increment and a migration function must run. Without a migration strategy, the first schema change will require users to manually clear their IndexedDB or face cryptic errors. The architecture specifies the schema but never addresses how it evolves.
Fix: Add a migration map to M0 (storage.js). When IDB opens, check stored version vs. current version. Run migration functions in sequence for any missed versions. Example: 
const MIGRATIONS = {
  1: (db) => { db.createObjectStore('scenarios', {keyPath:'id'}); },
  2: (db) => { db.createObjectStore('decision-journal', {keyPath:'id'}); },
  3: (db) => { /* add law_snapshot field */ }
};

No circular dependencies. The graph is a strict DAG (directed acyclic graph). Data flows: law-config → RE engines → CE engines → orchestrators (M2, M3) → UI modules. This direction is correct and should never be reversed.

One Improvement: RE7 and RE8 Should Not Import RE1–RE6

RE7 and RE8 are specified to "depend on RE1–RE6." But if they import those engines, they force recomputation. The better pattern: RE7 and RE8 receive pre-computed engine results as function parameters. M2 (the orchestrator) imports all engines and passes results between them. RE7 and RE8 have zero engine imports — they only receive data. This is already partially specified in the addendum ("via parameter, not import") but should be made explicit as a rule: orchestrators import engines; aggregators receive data.

Testing Strategy
F-03 No Test Architecture Defined HIGH
Each module prompt has 5–6 acceptance tests. These are end-state assertions, not a testing architecture. There is no unit test runner, no integration test strategy, no golden test cases for law change regression, and no strategy for testing UI modules. Without a test architecture, implementation will produce untestable code that is expensive to maintain.

Testing Architecture

Layer 1: Unit Tests (pure engine functions)

Runner: Vitest or Jest (both work without build toolchain via ESM). Run in Node.js. No browser needed. Test file: tests/unit/re3-aga.test.js alongside the engine file.

Coverage target: 100% of all RE and CE engine functions. Every edge case from F-02 error handling table above must have a corresponding unit test.

Test pattern:

// tests/unit/re3-aga.test.js
import { evaluateAGA } from '../../js/engines/re3-aga.js';
import { lawConfig } from '../../js/core/law-config.js';

describe('RE3 — Aggregate Gross Assets Engine', () => {
  test('cash only — below threshold', () => {
    const result = evaluateAGA({ asset_record: { cash: 5000000 } }, 'post_july4_2025');
    expect(result.total_aga).toBe(5000000);
    expect(result.exceeds).toBe(false);
    expect(result.status).toBe('pass');
  });
  test('goodwill FMV excluded from AGA', () => {
    const result = evaluateAGA({ asset_record: { cash: 5000000, goodwill_fmv: 50000000 } }, 'post_july4_2025');

  test('null asset_record returns insufficient_information', () => {
    const result = evaluateAGA({ asset_record: null }, 'post_july4_2025');
    expect(result.status).toBe('insufficient_information');

  // ... all edge cases

Review Area 9
Performance Review
F-07 Dashboard Performance at 400 Scenarios Without DC1 Cache MEDIUM
If ED1 (Founder Dashboard) reads all 400 scenarios and runs evaluateQSBS() on each to compute aggregate metrics, dashboard open time could exceed 2 seconds. Each evaluateQSBS() call runs 8 sub-engines. 400 × 8 = 3,200 sub-engine calls on dashboard open. Even at 1ms per call that is 3.2 seconds blocking the UI.
Fix: DC1 (Dashboard Cache) is recommended in the addendum but not formally specified. Specify it now: on every scenario save, write a lightweight summary object (status, confidence, aga_pct, holding_years, estimated_benefit) to a dedicated "dashboard-cache" IndexedDB key. Dashboard reads the cache only — never runs engines. Cache is invalidated on scenario save and on lawConfig change.
F-12 Offline Supabase Queue Size Not Bounded LOW
If a user works offline for a week with autosave firing every 500ms, the sync queue could grow to thousands of items. No queue size limit or flush strategy is defined.
Fix: Sync queue max size: 500 items. When limit is reached, collapse multiple saves of the same record into one (keep the most recent). On reconnect, flush in batches of 50 with 100ms gaps. Show "Syncing..." indicator during flush. Log failed items after 3 retries and surface them in Settings.

F-09 JSON Import Validation Underspecified MEDIUM
The architecture says "validate JSON against schema before writing." No schema is specified. An attacker could import a malicious JSON file that sets sec1202_basis to a very large number, causing the ten-times ceiling calculation to overflow. Or inject HTML into a string field that gets rendered unsanitized.
Fix: Specify the import validator in shared/validators.js. 
function validateImportJSON(data) {
  // 1. Must be a plain object (not an array, not a string)
  // 2. scenarios[] must be an array
  // 3. Each scenario: id must be a string, all numeric fields must be >= 0 and <= Number.MAX_SAFE_INTEGER
  // 4. All string fields: HTML-escape before storage (prevents XSS on render)
  // 5. Reject any field not in the canonical Scenario schema (unknown fields discarded, not rejected)
  // 6. law_snapshot.law_config_used must match one of the known law versions
  // 7. status must be one of the valid enum values
  // Returns: {valid: bool, errors: string[], sanitized_data: obj}
}

Overall Verdict

This architecture is production-ready after the 5 priority fixes listed in PR13. The core design decisions are correct: the dependency direction is right, the law configuration is properly centralized, the engine decomposition is appropriate for the domain complexity, and the extensibility design is strong.

The weaknesses are almost entirely in the cross-cutting concerns — error handling, result contracts, testing strategy — that were never fully specified because the architecture documents focused on module-level design. These are easy to fix now and very expensive to retrofit after 10,000 lines of code exist.

Do not start implementation until priority fixes P1 through P5 are formalized. Each takes less than one hour to specify. Each prevents weeks of future debugging.
