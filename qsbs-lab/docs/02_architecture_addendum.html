Architecture Refinement Addendum — v1.1

This document is a refinement pass only. It does NOT replace the base architecture documentation. It extends and decomposes it. All existing module numbers, file names, and architectural decisions in the base documentation remain valid. This addendum introduces 15 additional specialized engines and modules on top of the original 18, raises the total module count to 33, and formalizes standards that were implicit in the original.

Addendum Section 0
Overview & Change Log
What changed, what stayed the same, and why this refinement was needed.

What Stayed the Same

  • All 18 original modules (M0–M17) are unchanged in purpose, file location, and interface
  • The lawConfig structure and centralized law configuration remain as designed
  • The result object contract (status, confidence, reasonsFor, etc.) remains the canonical shape
  • The database schema is unchanged
  • The build order for the original 18 modules is unchanged
  • GitHub Pages deployment, Supabase sync, and local IndexedDB storage are unchanged

Addendum Section 1
Rules Engine Decomposition
The original rules-engine.js (M2) is decomposed into 8 specialized engines. M2 becomes an orchestrator that calls RE1–RE8 and assembles the final result object. Each RE engine is independently testable with no knowledge of the others.
M2 (rules-engine.js) still exists and still exports evaluateQSBS(scenario). Its internal implementation now delegates to RE1–RE8. All existing code that calls evaluateQSBS() continues to work unchanged. The decomposition is entirely internal.
RE1 — New
Acquisition Date Engine
Determines which law version applies based on the stock acquisition date. Handles edge cases: stock acquired on exactly July 4 vs July 5, 2025. Handles null dates. Returns law_version string and the full applicable law config block.
~80 linesZero dependenciesFile: js/engines/re1-acquisition-date.js

Export: determineAcquisitionDateLaw(acquisition_date)

Returns: {law_version, law_config_block, acquisition_date_parsed, note, source_citation}

Edge cases: null date → insufficient_information. July 4, 2025 → historical. July 5, 2025 → post_july4_2025. Future date → post_july4_2025 with note.

Acceptance test: determineAcquisitionDateLaw("2025-07-04").law_version === "historical" and determineAcquisitionDateLaw("2025-07-05").law_version === "post_july4_2025"

RE2 — New
Holding Period Engine
Computes holding years from acquisition date to exit date or exit year. Determines applicable exclusion percentage. Computes milestone dates (3yr, 4yr, 5yr). Determines if holding period is met for the applicable law version.
~120 linesDepends: RE1File: js/engines/re2-holding-period.js

Export: evaluateHoldingPeriod(acquisition_date, exit_date, law_version)

Returns: {holding_years, holding_days, exclusion_pct, milestone_3yr, milestone_4yr, milestone_5yr, days_to_next_milestone, next_milestone_benefit, pass, confidence, note, source_citation}

Critical: For post-2025 law, a 3-year hold produces 50% exclusion — not zero. Must never treat 3-year hold as failing.

RE3 — New
Aggregate Gross Assets Engine
Computes aggregate gross assets from the asset record. Compares to the applicable threshold. Determines pass/fail. Computes headroom. Alerts if within 20% of threshold. Handles controlled group assets and predecessor assets.
~150 linesDepends: RE1File: js/engines/re3-aga.js

Export: evaluateAGA(asset_record, law_version, controlled_group_assets, predecessor_assets)

Returns: {total_aga, components[], threshold, headroom, headroom_pct, exceeds, alert_level, pass, confidence, note, source_citation, valuation_paradox_note}

Valuation paradox note: Must include: "A higher FMV for contributed property increases the §1202 basis and ten-times ceiling, but also increases aggregate gross assets and may cause the threshold to be exceeded."

Critical: AGA = cash + adjusted tax bases of other property. NOT enterprise value. NOT FMV of all assets. Contributed property: use FMV per §1202(d)(2)(B). All other property: use adjusted tax basis.

RE4 — New
Original Issue Engine
Evaluates whether stock satisfies the original-issue requirement. Flags conversion scenarios as uncertain. Checks that consideration was money, eligible property, or qualifying services. Identifies prohibited modes of acquisition (secondary market, gift, inheritance).
~120 linesZero dependenciesFile: js/engines/re4-original-issue.js

Export: evaluateOriginalIssue(scenario)

Returns: {pass, confidence, consideration_type, conversion_detected, prohibited_acquisition, note, professional_questions[], source_citation}

Critical rule: Any conversion scenario (entity_path contains "convert", "revoke", "statutory", "reorganize") sets pass = null and confidence = "low". Never set pass = true for a conversion without noting professional review is required.

RE5 — New
Active Business Engine
Evaluates the 80%-by-value active business use test. Checks for passive investments, working capital excess, research and experimentation activities, software assets, subsidiary concerns, and excess non-business real estate. Reads from business_qual_result if available.
~180 linesDepends: M5 business-qual.jsFile: js/engines/re5-active-business.js

Export: evaluateActiveBusiness(scenario, business_qual_result)

Returns: {pass, confidence, active_use_pct_estimated, passive_asset_flags[], working_capital_concern, re_concern, subsidiary_concern, note, professional_questions[], source_citation}

If business_qual_result is null: return insufficient_information with note that business qualification interview must be completed first.

RE6 — New
Excluded Business Engine
Evaluates whether the business falls into an excluded category under §1202(e)(3). Scores each excluded category separately. Handles the reputation-or-skill catch-all. Returns a risk matrix with a score per category. BrightPath-aware: recognizes software delivery and automated curriculum as distinct from tutoring services.
~200 linesDepends: M5 business-qual.jsFile: js/engines/re6-excluded-business.js

Export: evaluateExcludedBusiness(scenario, business_qual_result)

Returns: {pass, confidence, risk_matrix[{category, risk_level, reasoning}], reputation_skill_risk, dominant_revenue_type, note, professional_questions[], source_citation}

Excluded categories scored separately: consulting, tutoring_as_service, reputation_or_skill, health, law, financial_services, and all others from §1202(e)(3). Each gets its own risk_level: none / low / medium / high / disqualifying.

BrightPath recognition: If software_orientation_score > 70 AND human_service_dependence < 20 AND automated_delivery = true → reduce excluded_business_risk materially. Document the reasoning explicitly.

RE7 — New
Qualification Decision Engine
The orchestrator. Receives results from RE1–RE6 plus C-Corp test and redemption check. Applies decision logic to produce the final qualification status and confidence level. Handles partial qualification. Handles the combination of multiple uncertain tests.
~150 linesDepends: RE1–RE6File: js/engines/re7-qualification-decision.js

Export: makeQualificationDecision(test_results{})

Decision logic:

if any test is "disqualifying" → does_not_qualify_under_stated_assumptions
if threshold.pass === false → does_not_qualify_under_stated_assumptions
if originalIssue.pass === null → uncertain_professional_review_required
if excludedBusiness.risk_level === "high" → uncertain_professional_review_required
if all tests pass or potentially_pass → likely_qualifies_under_stated_assumptions
if 1 test is uncertain → potentially_qualifies
if 2+ tests uncertain → uncertain_professional_review_required
never return a bare "qualifies" — always use the full qualified phrase

Returns: {status, confidence, overall_reasoning, combined_reasons_for[], combined_reasons_against[], snapshotted_law_version, source_citations[]}

RE8 — New
Explanation and Audit Engine
Assembles the complete human-readable explanation from all engine outputs. Produces the professional review packet section on qualification. Generates the calculation trace in display-ready format. Compiles all source citations. Generates professional review questions. This engine produces no new determinations — only explanation of determinations already made.
~180 linesDepends: RE1–RE7File: js/engines/re8-explanation.js

Export: buildExplanation(all_engine_results{})

Returns: {status_plain_language, confidence_explanation, qualification_matrix_rows[], calculation_trace_display[], source_register[], professional_questions[], plain_english_summary, disclaimer_text}

Critical: Every item in source_register must include: citation (e.g. "26 U.S.C. § 1202(d)"), proposition (what rule it supports), source_url, date_checked. No uncited legal conclusions permitted.

M2 as Orchestrator

The original rules-engine.js (M2) is retained as the public API. Its implementation becomes a thin orchestrator:

// js/engines/rules-engine.js (M2 — now an orchestrator)
import { determineAcquisitionDateLaw } from './re1-acquisition-date.js';
import { evaluateHoldingPeriod } from './re2-holding-period.js';
import { evaluateAGA } from './re3-aga.js';
import { evaluateOriginalIssue } from './re4-original-issue.js';
import { evaluateActiveBusiness } from './re5-active-business.js';
import { evaluateExcludedBusiness } from './re6-excluded-business.js';
import { makeQualificationDecision } from './re7-qualification-decision.js';
import { buildExplanation } from './re8-explanation.js';

export function evaluateQSBS(scenario) {
  const lawResult = determineAcquisitionDateLaw(scenario.acquisition_date);
  const holdingResult = evaluateHoldingPeriod(scenario.acquisition_date, scenario.exit_date, lawResult.law_version);
  const agaResult = evaluateAGA(scenario.asset_record, lawResult.law_version, scenario.controlled_group_assets, scenario.predecessor_assets);
  const originalIssueResult = evaluateOriginalIssue(scenario);
  const activeResult = evaluateActiveBusiness(scenario, scenario.business_qual_result);
  const excludedResult = evaluateExcludedBusiness(scenario, scenario.business_qual_result);
  const decision = makeQualificationDecision({law: lawResult, holding: holdingResult, aga: agaResult, originalIssue: originalIssueResult, active: activeResult, excluded: excludedResult});
  const explanation = buildExplanation({law: lawResult, holding: holdingResult, aga: agaResult, originalIssue: originalIssueResult, active: activeResult, excluded: excludedResult, decision});
  return { ...decision, ...explanation, individualTests: { acquisitionDate: lawResult, holdingPeriod: holdingResult, aga: agaResult, originalIssue: originalIssueResult, activeBusiness: activeResult, excludedBusiness: excludedResult } };
}
// Public API is identical to original M2. All callers unaffected.
Addendum Section 2
Calculation Engine Decomposition
The original calc-engine.js (M3) is decomposed into 7 specialized engines. M3 becomes an orchestrator. All existing callers are unaffected.
calc-engine.js (M3) still exports calcStockSaleTax(), calcAssetSaleTax(), calcExclusionLimit(), etc. Internally, it delegates to CE1–CE7. Backward compatible.
CE1 — New
Basis Engine
Computes and tracks all basis types. Normal shareholder basis, Section 1202 special basis, inside asset basis, contributed property FMV-based basis. Documents every step. Warns when bases differ. Explains the §1202(e)(3) FMV rule for contributed property.
~150 linesDepends: RE1File: js/engines/ce1-basis.js

Exports: computeNormalBasis(inputs), compute1202Basis(inputs), computeInsideBasis(asset_record), basisDifferenceWarning(normal, sec1202)

Critical output: When normal_basis !== sec1202_basis, output must include: "Warning: §1202 basis for the ten-times limitation differs from your normal shareholder basis. The ten-times limit uses the §1202 basis (FMV of contributed property), not the normal basis."

CE2 — New
Threshold Engine
Computes the applicable exclusion limitation. Fixed-dollar limit (accounting for prior exclusions). Ten-times-basis limit (using §1202 basis). Applicable limit (greater of the two). Documents every formula. Flags when prior exclusions reduce the fixed-dollar limit to zero.
~120 linesDepends: CE1, RE1File: js/engines/ce2-threshold.js

Export: computeExclusionLimit(sec1202_basis, prior_exclusions_used, law_version)

Returns: {fixed_dollar_base, prior_exclusions_used, fixed_dollar_remaining, ten_times_basis, applicable_limit, limiting_factor, trace[], source_citation}

limiting_factor: "fixed_dollar" | "ten_times" | "equal" — tells user which ceiling is controlling and why.

CE3 — New
Valuation Engine (Calculation Layer)
The calculation layer for all valuation methods. Distinct from the Valuation Engine in the original M4 (which handles UI and data). This engine performs the math: ARR multiple, revenue multiple, EBITDA multiple, DCF, comparables, NAV, IP-focused, custom. Returns weighted conclusion.
~200 linesZero dependenciesFile: js/engines/ce3-valuation.js

Note on relationship to M4: M4 (valuation.js) is the UI module that collects valuation inputs and displays charts. CE3 is the pure calculation engine that M4 calls. They are now explicitly separated. M4 depends on CE3. CE3 has no DOM knowledge.

Exports: calcARRMultiple(arr, multiples), calcEBITDAMultiple(ebitda, multiples), calcDCF(cash_flows, discount_rate, terminal), calcWeightedEV(methods, weights), calcEquityValue(ev, debt, cash)

CE4 — New
Stock Sale Engine
All calculations specific to a stock sale with QSBS treatment. Receives pre-computed exclusion limit from CE2. Computes amount realized, total gain, eligible gain, excluded gain, taxable gain, federal tax, NIIT, state tax, total tax, after-tax proceeds, effective rate. Returns with full trace.
~180 linesDepends: CE1, CE2, RE1, RE2File: js/engines/ce4-stock-sale.js

Export: calcStockSaleTax(scenario, exclusion_limit, exclusion_pct)

Returns: Full breakdown with every formula in trace[]. Source citations on each formula.

CE5 — New
Asset Sale Engine
All calculations specific to an asset sale. Corporate gain, corporate tax (21%), distributable proceeds, shareholder gain on distribution, shareholder tax, double-tax total, shareholder after-tax proceeds. Always includes the warning that QSBS does not protect corporate-level gain.
~150 linesDepends: CE1File: js/engines/ce5-asset-sale.js

Export: calcAssetSaleTax(scenario)

Required trace item: "QSBS exclusion under §1202 applies at the shareholder level to gain on the sale of qualifying stock. It does not protect the corporation from tax on gains recognized in an asset sale. The double-tax applies in full." — cited to 26 U.S.C. § 1202(a).

CE6 — New
Exit Tax Engine
The orchestrator for exit calculations. Calls CE4 or CE5 depending on sale type. Computes baseline (no QSBS). Computes old law vs. new law comparison. Computes partial qualification scenarios. Returns combined result object consumed by the Exit Calculator UI (M13).
~150 linesDepends: CE1–CE5, RE1, RE2File: js/engines/ce6-exit-tax.js

Export: calcExitTax(scenario) → {stock_sale, asset_sale, baseline, old_vs_new, savings, partial_qual_scenarios[]}

M3 becomes: A thin re-export layer that maintains backward compatibility by importing from CE6 and re-exporting the same function signatures.

CE7 — New
Sensitivity Analysis Engine
Runs parametric sensitivity analysis. Varies one or more inputs across a range and returns a result matrix. Runs the full rules engine + calc engine for each variation — not just the math, but also re-checks if qualification status changes at boundary values. Handles Monte Carlo simulation locally.
~200 linesDepends: CE6, RE7File: js/engines/ce7-sensitivity.js

Exports: runSensitivity(scenario, variables[]) → result matrix. runMonteCarlo(scenario, ranges{}, iterations) → distribution results.

Label as simulation: Monte Carlo output must always include: "This simulation is not a forecast or a legal probability estimate. It illustrates possible ranges of outcomes under stated assumptions."

BI1 Analysis Dimensions

Software vs. Service Revenue

Slider input: % from software subscriptions, % from automated diagnostic delivery, % from curriculum access, % from human services, % from consulting. Score updates in real time. Threshold: if software + automated > 70%, orientation is "software-dominant".

Founder Dependence

Questions: Does BrightPath require Kelly to deliver lessons? Does Kelly's reputation drive subscription renewals? Can the platform operate if Kelly is not involved for 30 days? Score 0-100. Low founder dependence = positive QSBS signal.

Tutor Dependence

Specific to BrightPath: Does delivery require human tutors? Do customers pay for tutor time or platform access? Can subscriptions scale without adding tutors? BrightPath's answer: no tutors in delivery chain. Document this explicitly with architectural evidence.

SaaS Characteristics

Recurring revenue model, monthly subscription, low marginal cost per additional user, platform not people, automated delivery, 24/7 availability without human intervention. BrightPath scores high on all. Document each characteristic as supporting evidence.

Proprietary IP Assessment

DCAR diagnostic engine, 5,000-entry misconception taxonomy, 17,524+ lesson atoms, 5-language curriculum library, parent script library, recalibration engine, prerequisite chain mapping. These are the principal assets. Value them relative to human capital.

Automation Score

Content generated once by AI pipelines and served deterministically. Zero AI at runtime. No tutors, no consultants, no founder in delivery chain. Engine-assembled lessons. Parent scripts pre-generated. This is the strongest QSBS positive signal in BrightPath's architecture.

Reputation-or-Skill Risk

Is BrightPath's principal asset Kelly's reputation? Is Kelly required for delivery? Can customers get the same value without Kelly? Current answer: platform is the product. Kelly built it but is not in the delivery chain. Document the distinction carefully.

Licensing Potential

Can BrightPath license curriculum to schools, districts, or other platforms? Licensing revenue is strongly software-oriented and reduces excluded-business risk. Document any existing or planned licensing arrangements.

Addendum Section 6 — New Behavior
Law Version Locking
A historical scenario must never silently recalculate under new law. This is one of the most important data integrity requirements in the application.

The Problem

If Congress amends §1202 in 2027 and the lawConfig is updated, all previously calculated scenarios would recalculate under the new law if only the law_version string is stored. A scenario calculated in June 2026 under post-July 4 2025 rules might show different results in 2028 when a user opens it — not because their inputs changed, but because the law changed.

The Solution: Law Snapshot

Every scenario permanently stores a complete snapshot of the lawConfig block used during its last calculation. This snapshot is immutable once saved. A scenario's results are always computed against its own stored law snapshot, not the current lawConfig.

// Added to scenarios table (database column: law_snapshot JSONB)
// Populated at calculation time, never updated
{
  law_version: "post_july4_2025",
  snapshot_date: "2026-06-15",
  law_config_used: {
    grossAssetsThreshold: 75000000,
    fixedDollarLimit: 15000000,
    exclusionSchedule: { 3: 0.50, 4: 0.75, 5: 1.00 },
    source: "One Big Beautiful Bill Act (2025)",
    dateChecked: "2026-06-01"
  },
  application_version: "1.1",
  calculated_at: "2026-06-15T14:30:00Z"
}

UI Behavior

  • If a user opens a historical scenario after lawConfig has been updated, show a banner: "This scenario was calculated using the law configuration from [snapshot_date]. The current law configuration differs. Click 'Recalculate' to update to current law, or 'View Historical' to review the original calculation."
  • Recalculate creates a new scenario_version snapshot before updating the law_snapshot
  • Historical view shows results with the original law snapshot, clearly labeled
  • The law_snapshot is never modified — a recalculation creates a new law_snapshot with the new law

Database Change

Add column to scenarios table: law_snapshot JSONB — nullable for scenarios created before this feature, populated on all new calculations. This is additive — no existing data is affected.

Every engine must produce a complete deterministic audit trail. This formalizes what was implicit in the original calculationTrace[] array.

Standard Audit Trail Shape

Every engine output must include an audit_trail object with these required fields:

// AuditTrail — required on every engine output
{
  engine: "re3-aga",                    // which engine produced this
  version: "1.0",                       // engine version
  calculated_at: "2026-06-15T14:30:00Z",
  inputs_used: {                        // complete copy of inputs — no references
    cash: 50000,
    equipment_basis: 10000,
    // ... all inputs
  },
  steps: [                              // calculation trace
    {
      step: 1,
      label: "Cash (included in AGA)",
      value: 50000,
      formula: "AGA += cash",
      source: "26 U.S.C. § 1202(d)(1)(A)"
    },
    // ...
  ],
  conclusions: [                        // what was determined
    {
      conclusion: "AGA does not exceed threshold",
      value: 60000,
      threshold: 75000000,
      pass: true,
      confidence: "high"
    }
  ],
  source_register: [                    // all legal sources applied
    {
      citation: "26 U.S.C. § 1202(d)",
      proposition: "Aggregate gross assets = cash + adjusted bases of other property",
      source_url: "https://uscode.house.gov/view.xhtml?req=granuleid:USC-prelim-title26-section1202",
      date_checked: "2026-06-01"
    }
  ],
  professional_questions: [            // questions generated by this engine
    "Were any assets contributed to the corporation? If so, the FMV at contribution (not carryover basis) is used for the AGA calculation."
  ],
  warnings: [],                        // any non-fatal warnings
  missing_inputs: []                   // inputs that were null or missing
}
The source_register in each engine is merged by RE8 (Explanation Engine) into the complete source register for the scenario. No engine should cite a rule without including the full citation details. No legal conclusion should appear without a source citation.

One Additional Decomposition Recommended

Dashboard Cache Layer (DC1)

Add js/engines/dc1-dashboard-cache.js: a lightweight cache engine that pre-computes dashboard metrics (aggregate AGA, combined scenario statuses, total estimated value, documentation scores) whenever a scenario is saved. Pattern: storage.save('scenarios', scenario) triggers dc1.recompute(company_id) which writes to a dashboard_cache key in IndexedDB. The dashboard reads from dashboard_cache only.
