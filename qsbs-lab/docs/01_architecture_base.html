Section 1
Executive Overview
Purpose, scope, intended users, and long-term vision for the BrightPath QSBS & Business Strategy Lab.

Purpose

The BrightPath QSBS & Business Strategy Lab is a strategic planning and educational tool to help business owners, founders, and their advisors model entity structures, Section 1202 QSBS eligibility, valuation scenarios, and tax-efficient exit paths. It is not a tax return preparer, legal opinion engine, or replacement for licensed professional counsel.

The primary user is Kelly O. Jakpa, founder of The BrightPath Education Group LLC, who needs to model the optimal entity and conversion path for BrightPath as it scales toward 5M+ users globally.

Scope

✅ In Scope
  • LLC, S-Corp, C-Corp modeling
  • Entity conversion analysis
  • Section 1202 QSBS eligibility
  • Post-July 4, 2025 law rules
  • Aggregate gross assets tracking
  • Basis tracking (shareholder, §1202, inside)
  • Exit tax calculators (stock sale, asset sale)
  • Valuation tracking (9 methods)
  • Business qualification interview
  • Up to 400 scenarios, multiple companies
  • Local (IndexedDB) + cloud (Supabase) storage
  • GitHub Pages deployment
  • Professional review packet export
❌ Out of Scope
  • Legal opinions or tax opinions
  • IRS filing or tax return preparation
  • Certified valuations or appraisals
  • Real-time market data feeds
  • Multi-user collaboration (v1)
  • AI inference for eligibility decisions
  • State-specific entity formation
  • International tax (v1)

Architectural Principles

🔒 Deterministic Rules

No AI inference decides eligibility. All results come from the configurable rule engine applied to user inputs. Every result is traceable to a specific rule and source citation.

📦 Module Independence

Each module is a self-contained JavaScript file with no circular dependencies. Modules communicate only through a central state bus and storage API.

🔌 Storage Abstraction

Application code never calls IndexedDB or Supabase directly. All persistence goes through a storage.js abstraction layer. Switching from local to cloud requires no application code changes.

📜 Law Config First

All tax law constants live in one lawConfig object. No legal constant appears hardcoded anywhere in the application. Updating the law requires editing one file.

🧮 Separated Concerns

UI rendering, calculation logic, rule evaluation, and storage are completely separate layers. A new UI can be built without touching the calculation engine.

📱 Offline First

The application works without internet. Local IndexedDB is the primary store. Supabase sync is additive. The app never blocks on network availability.

Module 2
Rules Engine (engines/rules-engine.js)
The deterministic QSBS eligibility engine. Takes a scenario object, applies lawConfig rules, returns a structured result object. Zero DOM. Zero storage. Never uses AI.
Critical PathHigh Difficulty~600 linesDepends on: M0

Key export: evaluateQSBS(scenario) → {status, confidence, reasonsFor[], reasonsAgainst[], missingInputs[], rulesApplied[], calculationTrace[], professionalReviewQuestions[], individualTests{}}

Status values: "likely_qualifies_under_stated_assumptions" | "potentially_qualifies" | "uncertain_professional_review_required" | "likely_does_not_qualify" | "does_not_qualify_under_stated_assumptions" | "insufficient_information"

Critical: Never hardcode thresholds — always read from lawConfig. Apply law by acquisition_date, not current date. Never return bare "qualifies".

Module 3
Calculation Engine (engines/calc-engine.js)
All tax mathematics. Computes eligible gain, exclusion amounts, fixed-dollar limit, ten-times-basis limit, federal tax, NIIT, state tax, total tax, after-tax proceeds. Handles stock sale and asset sale. Zero DOM.
Critical PathHigh Difficulty~800 linesDepends on: M0, M2

Key exports: calcExclusionLimit(scenario), calcExclusionPct(law_version, holding_years), calcStockSaleTax(scenario), calcAssetSaleTax(scenario), calcBaselineTax(gain, rates), calcOldVsNewLaw(scenario)

Critical distinction: ten_times uses sec1202_basis (FMV of contributed property), NOT normal stock_basis. These must NEVER be confused.

Module 4
Valuation Engine (engines/valuation-engine.js)
All 9 valuation methods. Weighted enterprise value conclusion. Separates enterprise value from equity value from aggregate gross assets — these must never be conflated.
Medium Difficulty~400 linesDepends on: M0

Key exports: calcWeightedEV(methods, weights), calcEquityValue(ev, debt, cash), calcAggregateGrossAssets(assetRecord), calcThresholdAnalysis(aga, law_version)

Critical: calcAggregateGrossAssets uses tax BASES not FMV (except contributed property per §1202(d)(2)(B)). Enterprise value is NEVER included in AGA.

Module 5
Business Qualification Engine (engines/business-qual.js)
Structured interview scoring. Evaluates revenue mix, delivery mechanism, founder dependence, asset composition, scalability. Returns scored indicators. Never classifies from a label — always from facts.
Medium Difficulty~500 linesDepends on: M0, M2

Key export: scoreBusinessQual(answers) → {classification[], indicators{software_orientation, human_service_dependence, consulting_risk, excluded_business_risk}, riskMatrix[], status, confidence, notes[]}

Modules 6-17

Dashboard, Company Wizard, Scenario Manager (400 scenarios, compare 6 side-by-side, autosave 500ms debounce), QSBS Wizard (6 steps), Stock Issuance Tracker, Valuation Tracker (charts), Exit Calculator, Entity Path Comparison, Evidence Tracker, Timeline, Reports, Settings.

Core Principles

  • Every user-owned table has id (UUID), user_id (FK to auth.users), created_at, updated_at
  • RLS ensures users can never read or write another user's data — enforced at database level, not application level

Section 7
State Management
Layer 1: In-Memory State

Active scenario, current company, UI state. Managed by state.js pub/sub bus. Lost on page reload — reconstructed from storage on init.

Layer 2: IndexedDB (Local)

All user data persisted locally. Works offline. Survives page reload. Holds up to 400 scenarios with full history. Schema versioned for migrations.

Layer 3: Supabase (Cloud)

Optional sync. Writes to both IndexedDB and Supabase on every save. On first load, fetches from Supabase and merges with local data using updated_at comparison.

Autosave Strategy

  • Every form field change triggers a debounced autosave (500ms delay)
  • Autosave writes to IndexedDB immediately — fast, sync-like
  • Supabase sync is queued and executed after autosave completes — async, non-blocking
  • Visible "Saved" / "Saving..." / "Unsaved changes" indicator in topbar

Conflict Resolution

  • Last-write-wins using updated_at timestamp when same scenario modified on two devices
  • Before overwriting, older version is saved to scenario_versions
  • User is notified and shown both versions with option to restore
